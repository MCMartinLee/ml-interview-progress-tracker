25-Day Sprint Detailed Guide

Use this when the tracker checklist is too compressed. Each day below explains where to look, what to create, what “done” means, and what to record.

How to use Week 1 ML Week 2 ML Week 3 LLM Week 4 Agents Templates

How to Use This Guide

For every tracker task, make one visible artifact: a notebook, markdown note, code file, solved problem, flashcard set, application log entry, or short reflection. If a resource link opens a repository, use the filename in the task as your anchor.

Reading

Read the named chapter or notebook, then write 5-8 bullets in your own words. Do not only watch or skim.

Coding

Create runnable files under prep/. A small working notebook beats a large unfinished one.

Cards

Create question-answer prompts in Anki, markdown, Notion, or a spreadsheet. Review old cards before making new ones.

Log

End each day with what shipped, what broke, what confused you, and the next blocker.

Core Folder Meanings

  • prep/ml: classical ML notebooks, notes, metrics, charts, and model comparisons.
  • prep/llm: tokenization, attention, GPT components, training, and fine-tuning practice.
  • prep/agents: tool use, RAG, memory, planning, guardrails, and job-fit agent code.
  • prep/system_design: architecture docs, diagrams, estimates, and ML system design notes.
  • prep/leetcode: problem solutions, tests, complexity notes, and repeated mistakes.
  • prep/behavioral: intro pitch, STAR stories, ownership examples, and follow-up answers.
  • prep/job_search: roles, resume variants, job descriptions, contacts, and application outcomes.

Week 1: Classical ML Foundations

Day 1: Setup and ML Landscape

Primary source: Hands-On ML 01_the_machine_learning_landscape.ipynb and Chapter 1 concepts.

  • Read for: supervised, unsupervised, semisupervised, self-supervised, reinforcement learning, batch vs. online, instance-based vs. model-based, generalization, leakage, bias, variance.
  • Create prep/resources.md with links to Hands-On ML, LLMs-from-scratch, nanoChat, system-design-primer, Microsoft agents, job boards, and interview question resources.
  • Create the prep folder structure and a short README in each folder.
  • Create prep/ml/notebooks/day01_setup_and_eda.ipynb. EDA means exploratory data analysis: inspect shape, columns, missing values, examples, and summary statistics.
  • Create 15 cards using the prompts in the Day 1 markdown backup or copy them into Anki.
  • Done means: folders exist, resources are saved, setup notebook opens, 15 cards exist, and the work is committed or logged.
Day 2: End-to-End ML Pipeline
  • Read Hands-On ML 02_end_to_end_machine_learning_project.ipynb. Focus on train/test split, stratified split, missing values, categorical encoding, scaling, pipelines, and cross-validation.
  • Build a notebook under prep/ml/notebooks/day02_pipeline.ipynb that creates numeric and categorical preprocessing with Pipeline and ColumnTransformer.
  • Train one simple baseline. If the task says logistic regression but the dataset is regression, use linear regression or logistic regression on a classification dataset and note the change.
  • Save a small metric table. Classification: accuracy, precision, recall, F1, ROC-AUC. Regression: MAE, RMSE, R2.
  • Cards: stratification, leakage, imputation, one-hot encoding, scaling, grouped split, time split.
  • Done means: the pipeline runs top to bottom and you can explain how leakage was avoided.
Day 3: Classification and Evaluation
  • Read Hands-On ML 03_classification.ipynb. Focus on confusion matrix, precision, recall, F1, threshold tuning, ROC, PR curve, multiclass, and error analysis.
  • Create prep/ml/notebooks/day03_classification_metrics.ipynb with dummy, logistic regression, and random forest models.
  • Plot or print confusion matrices, ROC curves, and PR curves. Change a decision threshold and record what improves and what gets worse.
  • Write 5 bullets explaining when to choose precision, recall, F1, ROC-AUC, or PR-AUC.
  • LeetCode: include the invariant for the sliding window solution, not just code.
  • Done means: you can explain false positives and false negatives using a concrete business example.
Day 4: Linear Models and Week 1 Deliverable
  • Read Hands-On ML 04_training_linear_models.ipynb. Focus on gradient descent, polynomial regression, regularization, logistic regression, and softmax.
  • Create prep/ml/notebooks/day04_linear_models.ipynb. Compare at least three regularization settings and plot validation performance.
  • Add a Week 1 README under prep/ml/README.md with links to notebooks, one result table, and 5 interview talking points.
  • Cards: gradient descent, learning rate, L1 vs. L2, regularization strength, softmax, logistic regression output.
  • Done means: Week 1 ML work is explainable without opening every notebook.
Day 5: Week 1 Review and Buffer
  • Do not add lots of new work. Finish broken notebooks, rerun them from top to bottom, and fix imports or paths.
  • Closed-book explain: supervised learning, leakage, bias/variance, regularization, and model evaluation.
  • Review every due card and rewrite failed answers into shorter, sharper cards.
  • Redo one coding problem from Week 1 and record the mistake you are most likely to repeat.
  • Done means: Week 1 has a clean README, runnable notebooks, at least 50 total cards, and a written blocker list.

Week 2: Trees, Ensembles, Unsupervised Learning, and PyTorch

Day 6: Decision Trees
  • Read Hands-On ML 05_decision_trees.ipynb. Focus on CART, Gini, entropy, tree depth, leaf constraints, regression trees, and instability.
  • Train one unrestricted tree and one regularized tree. Compare train vs. validation performance to show overfitting.
  • Save a tree plot or feature-importance chart, plus a short caveat about interpretability.
  • Done means: you can explain why trees overfit and which hyperparameters reduce it.
Day 7: Ensembles
  • Read Hands-On ML 06_ensemble_learning_and_random_forests.ipynb. Focus on voting, bagging, random forests, AdaBoost, gradient boosting, and stacking.
  • Use identical train/validation or cross-validation splits across models so the comparison is fair.
  • Create a model-comparison table and choose a champion model with one reason and one caveat.
  • Done means: you can explain bagging vs. boosting and why stacking can leak if built carelessly.
Day 8: Dimensionality Reduction
  • Read Hands-On ML 07_dimensionality_reduction.ipynb. Focus on curse of dimensionality, PCA, explained variance, randomized PCA, and manifold learning.
  • Standardize features before PCA. Fit PCA retaining 95% variance and plot cumulative explained variance.
  • Compare model speed and accuracy before and after PCA.
  • Done means: you can say when PCA helps and when it harms interpretability.
Day 9: Unsupervised Learning
  • Read Hands-On ML 08_unsupervised_learning.ipynb. Focus on K-means, inertia, silhouette score, GMM, anomaly detection, and limitations.
  • Compare several values of K. Record inertia and silhouette score rather than trusting one plot.
  • Fit a GMM and visualize clusters after PCA if the feature dimension is high.
  • Done means: you can explain why unsupervised evaluation is less direct than supervised evaluation.
Day 10: Artificial Neural Networks
  • Read Hands-On ML 09_artificial_neural_networks.ipynb. Focus on perceptrons, MLPs, activations, output layers, losses, and backpropagation.
  • Implement a small two-layer PyTorch MLP with shape assertions and train/validation loss logging.
  • Compare ReLU and sigmoid behavior. Note vanishing gradients and dying ReLUs.
  • Done means: your model trains and you can explain input shape, output shape, loss, and optimizer.
Day 11: PyTorch and Week 2 Deliverable
  • Read Hands-On ML 10_neural_nets_with_pytorch.ipynb and skim 11_training_deep_neural_networks.ipynb.
  • Create reusable PyTorch functions: dataset/dataloader, train_one_epoch, evaluate, checkpoint save/load, early stopping, seed setting.
  • Add a README usage example that shows how to run training.
  • Done means: Week 2 has a reproducible training loop and at least 100 cumulative cards.
Day 12: Week 2 Review and Buffer
  • Rerun all classical ML notebooks and the PyTorch loop. Fix nondeterminism and broken paths.
  • Explain trees, forests, boosting, PCA, K-means, GMM, MLPs, and backpropagation without notes.
  • Run a 20-card random recall test and rewrite answers longer than four sentences.
  • Done means: your Week 2 artifacts are runnable and your weakest three ML topics are named.

Week 3: LLM Fundamentals

Day 13: Tokenization and Data Loading
  • Read LLMs-from-scratch Chapter 2. Focus on tokenization, vocabulary, token IDs, sliding windows, embeddings, positions, and next-token targets.
  • Build a tiny corpus and print input/target windows. Verify that the target is shifted by one token.
  • Done means: you can explain how raw text becomes training examples.
Day 14: Attention
  • Read Chapter 3. Focus on Q/K/V, scaled dot-product attention, causal masks, padding masks, dropout, and multi-head attention.
  • Implement single-head attention, then multi-head attention. Print tensor shapes at each step.
  • Test that future tokens receive zero attention after applying a causal mask.
  • Done means: you can draw the attention shape flow from memory.
Day 15: GPT Architecture
  • Read Chapter 4. Focus on transformer blocks, residual paths, layer norm, FFN, token embeddings, positional embeddings, parameter count, and generation.
  • Run a forward pass on dummy IDs and verify output shape: batch, sequence length, vocabulary size.
  • Implement greedy generation and write down where temperature or top-k would fit.
  • Done means: you can trace token IDs to logits and generated tokens.
Day 16: Pretraining and nanoChat
  • Read Chapter 5 and inspect nanoChat README, tokenizer, GPT model, and training pipeline.
  • Run a tiny controlled training job or inspect the code path if compute is limited. Record train loss, validation loss, checkpoint behavior, and one generated sample.
  • Create a pipeline diagram from raw text to tokenizer to batches to model to checkpoint to generation.
  • Done means: you can explain next-token cross-entropy and validation loss.
Day 17: Fine-Tuning and Research Deep Dive
  • Read Chapter 6. Compare feature extraction, partial fine-tuning, full fine-tuning, freezing, and output-head replacement.
  • Run reduced classification fine-tuning or trace the notebook with small data. Count trainable parameters before and after freezing.
  • Record overfitting risks and three deep-dive questions you would ask in an interview.
  • Done means: you can distinguish pretraining, classification fine-tuning, and feature extraction.
Day 18: Instruction Fine-Tuning and Week 3 Deliverable
  • Read Chapter 7. Focus on instruction formatting, masks, SFT datasets, evaluation, inference, and chat templates.
  • Format five instruction examples and build or inspect an instruction dataset loader.
  • Add tests for tensor shapes, causal masking, and generation termination.
  • Done means: Week 3 has an LLM README, 150 cumulative cards, and a clear story from tokenization to SFT.
Day 19: Week 3 Review and Buffer
  • Run LLM tests, verify causal masking and shapes, reload a checkpoint, and generate a small sample.
  • Explain tokenizer to logits to loss to generation without notes.
  • Move failed cards into a Top 20 Before Interview deck.
  • Done means: your LLM weak areas are explicit and your code path is rerunnable.

Week 4: Agents, RAG, Memory, and ML System Design

Day 20: Agent Fundamentals and Tool Use
  • Read the Microsoft agent intro, frameworks, design patterns, and tool-use lessons. Focus on goal, state, observation, action, tool, policy, utility, and guardrails.
  • Create prep/agents/job_fit_agent with mock resume, job-description, skill-extraction, and overlap tools.
  • Use deterministic routing first. Avoid adding an LLM until the workflow is testable.
  • Done means: three test job descriptions produce predictable outputs.
Day 21: Agentic RAG
  • Read the Microsoft Agentic RAG lesson. Focus on query planning, retrieval, grounding, tool selection, response generation, and evaluation.
  • Store five job descriptions, chunk responsibilities and qualifications, and implement simple retrieval.
  • Output matched skills, missing skills, evidence snippets, and resume suggestions with sources.
  • Done means: every recommendation links back to retrieved evidence.
Day 22: Trust, Planning, and Multi-Agent Patterns
  • Read trustworthy agents, planning, multi-agent, and metacognition lessons. Focus on human approval, planner/executor/evaluator separation, max steps, and failure modes.
  • Add planner, executor, and evaluator steps to the job-fit agent.
  • Require human approval before generating resume replacement text and test tool failures or repeated loops.
  • Done means: the agent has limits, approval points, and tests for bad inputs.
Day 23: Protocols, Context, and Memory
  • Read Microsoft lessons on protocols, context engineering, and memory. Focus on MCP, A2A, context budgets, short-term memory, long-term memory, contamination, and compression.
  • Define a context schema separating profile, current job, evidence, and temporary state.
  • Add a length budget, summary memory, reset command, and tests preventing cross-company contamination.
  • Done means: memory cannot leak one company's details into another company's answer.
Day 24: ML System Design
  • Read system-design-primer sections on latency, throughput, availability, consistency, replication, sharding, caches, queues, and estimates.
  • Create prep/system_design/realtime_ml_inference.md with use case, constraints, API, online path, batch path, preprocessing, model service, cache, queue, monitoring, versioning, rollback, failure modes, latency budget, and capacity estimates.
  • Draw an architecture diagram. A simple Mermaid diagram is enough.
  • Done means: you can explain online inference vs. batch jobs and where drift monitoring fits.
Day 25: Final Integration and Deliverable
  • Run all notebooks top to bottom, remove large outputs and secrets, and make sure README links work.
  • Create a root README linking ML, LLM, agent, system-design, LeetCode, behavioral, and job-search artifacts.
  • Add result tables, screenshots, talking points, limitations, and next steps.
  • Run closed-book recall: 10 ML, 10 LLM, and 10 agent/system cards. Score immediate, partial, or failed.
  • Done means: someone reviewing your prep repo can see what you built, what you learned, and what you would discuss in interviews.
